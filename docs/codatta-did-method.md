# Codatta DID Method Specification

**DID Method Name:** `codatta`

## Abstract

`did:codatta` is a Decentralized Identifier (DID) method designed for Web3
applications. It aims to provide a unified, verifiable, and controllable digital
identity for users, data assets, and smart contracts.

Unlike DID methods that derive the method-specific identifier from an on-chain
address, `did:codatta` derives it from a random value. This keeps the identifier
chain-independent and makes the method straightforward to deploy across multiple
chains. Document updates are authorized through `controller` entries, and the
DID's own identifier always retains the highest authority over its document,
making identity control both safer and more flexible. The method supports
on-chain authentication, cross-chain interaction, and credential management,
providing a trustworthy foundation for data access control, asset management,
and decentralized applications.

## DID Method Name

The method name that identifies this DID method is `codatta`.

A DID using this method **MUST** begin with the prefix `did:codatta`. Per DID
Core, this prefix is lowercase.

## Identifier Syntax

A `did:codatta` identifier has the form:

```
did:codatta:<method-specific-id>
```

The `<method-specific-id>` is the UUID-formatted rendering of a 128-bit unsigned
integer (`uint128`): five groups of hexadecimal digits separated by hyphens in
the `8-4-4-4-12` layout.

```abnf
codatta-did        = "did:codatta:" method-specific-id
method-specific-id = 8hex-digit "-" 4hex-digit "-" 4hex-digit "-" 4hex-digit "-" 12hex-digit
hex-digit          = DIGIT / "A" / "B" / "C" / "D" / "E" / "F"
                           / "a" / "b" / "c" / "d" / "e" / "f"
```

Example:

```
did:codatta:95228308-9d75-4dd8-8958-2713b92d3d71
```

The `did:codatta:` prefix is matched case-sensitively. The hexadecimal section
accepts both upper- and lower-case digits. The reference generator produces a
strict UUIDv4 (it sets the version and variant bits), but because the underlying
value is an arbitrary `uint128`, resolvers validate the UUID *layout* only and do
not require the version/variant nibbles of a strict UUIDv4; any value conforming
to the layout is accepted. To avoid two distinct string forms for the same DID,
resolvers **MUST** emit the identifier in canonical lowercase
([RFC 4122](https://www.rfc-editor.org/rfc/rfc4122) §3); the reference resolver
does so.

## Method-Specific Identifier

The `<method-specific-id>` is generated by a random algorithm and has the data
type `uint128`.

It is **RECOMMENDED** to generate the value as a UUIDv4. A UUIDv4 has 122
effective random bits, but using the UUID format improves readability and
conforms to the UUID conventions. An implementation **MAY** store the value
on-chain as a `uint128`; the DID resolver converts it to the UUID string format
when resolving.

## DID Document

A `did:codatta` DID Document is a JSON object. The following properties are
defined.

### Required Properties

| Property             | Description                                                                                          |
| -------------------- | ---------------------------------------------------------------------------------------------------- |
| `@context`           | DID Document context, included for compatibility with the W3C DID JSON-LD representation. An array.   |
| `id`                 | The DID identifier string.                                                                            |
| `owner`              | Wallet address — for example, an Ethereum address if deployed on Base Chain.                          |
| `controller`         | **MUST** contain at least the DID's own identifier. An array.                                          |
| `verificationMethod` | **MUST** contain at least one verification method that is verifiable on the deployment chain. An array.|

### Optional Properties

| Property                | Description                                                                                              |
| ----------------------- | -------------------------------------------------------------------------------------------------------- |
| `alsoKnownAs`           | Aliases for the DID subject, expressed as an array of URIs (per DID Core).                                |
| `authentication`        | References to `verificationMethod` entries used for authentication (an array of verification method `id` strings). |
| `assertionMethod`       | Verification methods used for signing assertions. An array.                                              |
| `keyAgreement`          | Verification methods used for encrypted communication. An array.                                         |
| `capabilityInvocation`  | Verification methods used to invoke capabilities. An array.                                              |
| `capabilityDelegation`  | Verification methods used to delegate capabilities. An array.                                            |
| `service`               | Service endpoints; each item contains `id`, `type`, `serviceEndpoint`, etc. An array.                    |

The verification relationships (`authentication`, `assertionMethod`,
`keyAgreement`, `capabilityInvocation`, `capabilityDelegation`) are expressed as
arrays of references to `verificationMethod` entries (verification method `id`
strings); this method does not embed inline verification methods in a
relationship.

> **Method-specific extension properties.** `owner` is a `did:codatta`
> extension property, not a property defined by W3C DID Core. Likewise, the
> custom `verificationMethod.type` values defined in
> [Verification Methods](#verification-methods) (`EmailChallenge`, `GoogleAuth`,
> `EthereumAddress`, `TonAddress`) and their value fields (`emailAddress`,
> `googleOpenID`, `ethereumAddress`, `tonAddress`) are method-specific. The
> `@context` shown below includes only `https://www.w3.org/ns/did/v1`, under
> which these terms are undefined; a deployment that needs the document to be
> valid JSON-LD **SHOULD** also include a method-specific `@context` that defines
> them.

### Example

```json
{
  "@context": ["https://www.w3.org/ns/did/v1"],
  "id": "did:codatta:95228308-9d75-4dd8-8958-2713b92d3d71",
  "owner": "0x1234567890abcdef1234567890abcdef12345678",
  "controller": ["did:codatta:95228308-9d75-4dd8-8958-2713b92d3d71"],
  "verificationMethod": [
    {
      "id": "did:codatta:95228308-9d75-4dd8-8958-2713b92d3d71#key-1",
      "type": "EthereumAddress",
      "controller": "did:codatta:95228308-9d75-4dd8-8958-2713b92d3d71",
      "ethereumAddress": "0x1234567890abcdef1234567890abcdef12345678"
    }
  ]
}
```

## Verification Methods

In addition to standard verification method types, `did:codatta` defines the
following method-specific types. For each type it is **RECOMMENDED** to obscure
the identifying value with a salted hash (`hash + salt`) to avoid leaking private
information.

### EmailChallenge

Verifies control of an email address through a challenge sent to it (for example,
a verification code).

```json
{
  "id": "did:codatta:<id>#key-1",
  "type": "EmailChallenge",
  "controller": "did:codatta:<id>",
  "emailAddress": "name@email.com"
}
```

It is **RECOMMENDED** to store `emailAddress` as a salted hash.

### GoogleAuth

```json
{
  "id": "did:codatta:<id>#key-1",
  "type": "GoogleAuth",
  "controller": "did:codatta:<id>",
  "googleOpenID": "1234"
}
```

It is **RECOMMENDED** to store `googleOpenID` as a salted hash.

### EthereumAddress

Compatible with existing Ethereum wallet login.

```json
{
  "id": "did:codatta:<id>#key-1",
  "type": "EthereumAddress",
  "controller": "did:codatta:<id>",
  "ethereumAddress": "0x1234567890abcdef1234567890abcdef12345678"
}
```

It is **RECOMMENDED** to store `ethereumAddress` as a salted hash.

### TonAddress

Compatible with existing TON wallet login.

```json
{
  "id": "did:codatta:<id>#key-1",
  "type": "TonAddress",
  "controller": "did:codatta:<id>",
  "tonAddress": "EQABIjRWeJq83vASNFZ4mrze8BI0VniavN7wEjRWeJq83iSU"
}
```

It is **RECOMMENDED** to store `tonAddress` as a salted hash.

## Operations

Operations are defined here in terms of their inputs, the authorization and
processing they require, and their outputs. They are described abstractly so the
method remains chain-independent: the concrete on-chain binding — which chain a
deployment uses, the registry contract, and the transaction encoding — is
**deployment-specific and out of scope of this specification**. In a deployment,
write operations are submitted as transactions to an on-chain DID registry that
stores and verifies permissions; an off-chain resolver indexes the resulting
state and serves the assembled DID Document (see [Storage Model](#storage-model)).

### Create (Register)

- **Input:** the caller's account. The `method-specific-id` is **not** supplied
  by the caller — it is generated during registration (see
  [Method-Specific Identifier](#method-specific-identifier)).
- **Authorization:** registration is performed through a *registrar*. The
  underlying registry accepts `register` only from an allow-listed registrar
  contract; end users call the registrar, which generates the identifier and
  assigns ownership to the caller. The set of registrars is governed by the
  registry administrator.
- **Processing:** the registrar generates a new `uint128` identifier, and the
  registry records it with the caller as `owner`. The owner holds controller
  authority over its own DID (see [Update](#update)). A registrar **MAY** also,
  in the same flow, add initial verification methods and authentication entries
  before assigning ownership to the user.
- **Output:** a resolvable DID owned by the caller.

It is **RECOMMENDED** that registrars impose a limit to prevent large-scale
squatting of identifiers — for example, requiring registering users to pay a
cost.

### Read (Resolve)

- **Input:** a `did:codatta` DID.
- **Processing:** the resolver validates the identifier syntax, looks up the
  current registry state for that DID, and assembles a W3C DID Document. The
  on-chain registry additionally exposes the document for direct, highest-trust
  reads.
- **Output:** a [DID Resolution Result](https://w3c.github.io/did-resolution/) —
  `didDocument`, `didResolutionMetadata`, and `didDocumentMetadata` — or, when the
  client requests a bare document representation, the DID Document itself.

The off-chain resolver implements the
[W3C DID Resolution](https://w3c.github.io/did-resolution/) HTTP(S) binding:
`GET /1.0/identifiers/{did}`, with `Accept`-based content negotiation between the
bare DID Document (`application/did+ld+json` / `application/did+json`) and the
full DID Resolution Result (`application/ld+json;profile="https://w3id.org/did-resolution"`).
Resolution errors are reported via HTTP status and `didResolutionMetadata.error`
(for example, *notFound* for an unregistered DID, *invalidDid* for a malformed
identifier).

### Update

- **Input:** the target DID, the operating DID (`operator`), the property to
  change, and the new value(s).
- **Authorization:** the `operator` **MUST** be a `controller` of the target DID.
  Separately, ownership confers a built-in control capability: the DID owner is
  always authorized to operate on its own DID. This capability is inherent to
  ownership rather than a stored `controller` entry, so it cannot be removed and
  an owner retains authority over the document regardless of the `controller`
  contents.
- **Processing:** the registry applies one of the following changes:
  - set or revoke a key–value attribute;
  - add a single item to, or revoke a single item (by index) from, an array
    attribute (`@context`, `verificationMethod`, `alsoKnownAs`, `authentication`,
    `assertionMethod`, `keyAgreement`, `capabilityInvocation`,
    `capabilityDelegation`, `service`);
  - add or revoke a `controller`.

  An array attribute is mutated one item at a time — an individual item is the
  smallest updatable unit; there is no in-place element replacement (replace a
  whole value by revoking the old item and adding the new one).
- **Output:** the updated DID Document.

### Transfer

- **Input:** the target DID and the new `owner` address.
- **Authorization:** only the current DID owner.
- **Processing:** the registry changes `owner` to the new address.
- **Output:** the DID Document with the updated `owner`. Transferring ownership
  can be used to extend the lifecycle of a DID.

### Deactivate

Deactivation is **not currently supported**. The on-chain DID registry exposes no
deactivation operation, so a `did:codatta` DID cannot presently be marked
inactive. Owner-initiated deactivation is anticipated as future work; this
specification will define its inputs, authorization, and processing once the
mechanism is implemented.

## Storage Model

`did:codatta` does not restrict where the DID Document is stored, but a combined
on-chain and off-chain storage model is **RECOMMENDED**.

### On-Chain Storage

All write operations are initiated on-chain. The chain **MUST** store and verify
the user's permission information and **MUST** expose the stored DID Document for
direct reads, providing DID Document data at the highest level of trust. (The
resolver that serves the W3C DID Document over HTTP is the off-chain component
described in [Off-Chain Storage](#off-chain-storage) and [Resolve](#read-resolve).)

### Off-Chain Storage

To provide a complete query service and a better user experience, the off-chain
layer stores the full history of DID operations along with the latest state. The
on-chain state is authoritative: the off-chain layer is an index of on-chain
events and **MUST NOT** diverge from on-chain truth. On any discrepancy, the
on-chain registry prevails.

## Security Considerations

This section addresses the security considerations required of a DID method
specification by [DID Core §8.3](https://www.w3.org/TR/did-core/#security-requirements).

- **Update authorization.** A DID Document may be modified only by an account
  that owns a `controller` of the target DID; ownership is verified on-chain per
  operation. The DID owner additionally holds a built-in control capability over
  its own DID — inherent to ownership rather than a stored `controller` entry, so
  it cannot be removed. An owner therefore retains authority over its document
  regardless of the `controller` set, and a DID cannot be locked out of its own
  document by controller changes.
- **Verifiable data registry integrity.** All write operations are initiated
  on-chain, and the chain stores and verifies permission information. The
  integrity and availability of a DID Document therefore inherit the security
  properties of the deployment chain.
- **Cryptographic agility.** Verification is expressed through the extensible
  `verificationMethod` and verification-relationship properties, allowing new
  verification method types to be adopted without changing the method itself.
- **Namespace abuse.** It is **RECOMMENDED** to impose a registration cost (see
  [Create](#create-register)) to mitigate large-scale squatting of identifiers.

## Privacy Considerations

This section addresses the privacy considerations required of a DID method
specification by [DID Core §8.4](https://www.w3.org/TR/did-core/#privacy-requirements).

- **Minimize personal data.** Identifying values carried in verification methods
  (email address, Google OpenID, wallet addresses) **SHOULD** be stored as
  salted hashes rather than in the clear, so that publishing a DID Document does
  not disclose the underlying personal data.
- **Reduced address correlation.** Because the method-specific identifier is a
  random value rather than a derivation of an on-chain address, the identifier
  itself does not correlate a DID with a particular wallet address.
- **On-chain permanence.** Data written on-chain is effectively permanent and
  publicly readable. Implementers **SHOULD NOT** place plaintext personal data in
  on-chain storage, and **SHOULD** keep correlatable or revocable information in
  the off-chain layer where appropriate.
