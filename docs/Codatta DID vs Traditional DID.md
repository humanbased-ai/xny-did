# Codatta DID 与传统去中心化身份方案的全面对比

## 标识符设计

传统 DID 方案（如 did:ethr、did:pkh）直接使用区块链钱包地址作为标识符，身份与特定链上的特定账户强绑定。用户换钱包就意味着换身份，丢失私钥就意味着永久丢失身份。

Codatta DID 使用链上随机生成的 UUIDv4 作为标识符（格式为 `did:codatta:<uuid>`），与任何钱包地址、链环境完全解耦。用户可以转移 DID 的所有权到新地址，身份本身不会因为钱包变更而消失。这也使得同一个身份天然可以在多条链上使用，不需要为每条链维护一个独立的 DID。

## 所有权与授权模型

传统方案通常采用简单的 owner 模型——谁控制这个地址的私钥，谁就拥有该 DID 的全部权限，没有中间层级，也无法将部分权限委托给其他身份。

Codatta DID 引入了控制器（controller）机制。DID owner 拥有最高权限，同时可以添加其他 DID 作为 controller，由 controller 代表 owner 执行 DID Document 管理操作。这使得组织场景下的权限委托成为可能——比如一个企业 DID 可以授权多个员工 DID 作为 controller，而不需要共享同一把私钥。controller 可以随时被撤销，因此在安全性和可管理性上优于单一私钥直接共享。需要注意的是，当前 controller 模型主要提供“是否可管理该 DID”的权限划分，还不是按字段或按角色划分的细粒度授权模型。

## 数据模型与属性扩展性

传统方案的合约实现通常只覆盖了 W3C DID Core 规范中的标准字段，没有提供链上自定义属性的能力。如果开发者需要为身份附加业务特定的数据，往往只能通过 service endpoint 间接引用外部存储，增加了系统复杂度和对链下服务的依赖。

Codatta DID 同样遵循 W3C DID Core 规范，在合约层原生提供标准字段（verificationMethod、authentication、assertionMethod、keyAgreement、capabilityInvocation、capabilityDelegation、alsoKnownAs、service）的存储支持，同时支持用户自定义的 KV 属性。任何不与系统保留字段冲突的键值对都可以直接写入链上，成为 DID Document 的一部分。开发者无需修改合约，即可为身份附加业务特定的元数据。

## 验证方法的多样性

传统方案的验证方法通常局限于链原生的密码学体系。例如 did:ethr 主要支持 secp256k1 签名验证，与以太坊生态绑定。

Codatta DID 的验证方法设计为可扩展的，目前已定义支持 EthereumAddress、TonAddress、EmailChallenge、GoogleAuth 等多种类型。这意味着一个 Codatta DID 可以同时关联以太坊钱包、TON 钱包、邮箱、Google 账号等多种身份凭证，实现 Web2 和 Web3 身份的统一聚合。新的验证方法类型可以通过添加数组属性的方式引入，不需要升级合约。

## 隐私保护

传统方案的验证方式主要限于钱包地址和公钥，涉及的隐私数据类型相对单一，链上暴露的也主要是这些链原生信息。

Codatta DID 由于支持了更丰富的身份凭证类型（邮箱、Google 账号、多链钱包地址等），天然会涉及更多隐私敏感数据。因此在设计上，Codatta DID 推荐对这些敏感标识信息使用 hash+salt 方式存储。链上只保存哈希值，验证时由验证方提供原文和盐值进行比对。这在保证可验证性的同时，避免了身份信息的明文泄露，降低了用户被追踪和关联分析的风险。

## 数据生命周期管理

传统方案对数据的删除处理方式不一。有些方案直接从链上移除数据，有些则不支持删除。

Codatta DID 对不同类型的数据采用不同的撤销策略。数组属性在撤销时不会从链上移除，而是被标记为 `revoked`；KV 属性撤销后则会从当前状态中删除。两类操作都会在链上留下事件记录，因此历史变更仍然可以被审计和追溯。对于数组属性，DID Resolver 在构建 DID Document 时可以自动过滤已撤销的项，对上层应用透明。这种设计兼顾了当前状态查询效率与历史审计需求。

## 架构分层与性能

部分链上 DID 实现的读取过程仍然高度依赖链上查询或频繁 RPC 调用，因此在吞吐量、延迟和查询体验上容易受到区块链基础设施的限制。

Codatta DID 采用四层架构：链上合约层（DIDRegistry + DIDRegistrar）负责数据存储和权限控制；链下 Indexer 层监听链上事件并同步到高性能数据库；DID Resolver 层提供标准的 DID 解析接口；应用层（登录、授权等）直接调用 Resolver。写操作走链上确保安全性和不可篡改性，读操作走链下确保高性能和低延迟。重要的是，链上存储的是完整数据，链下索引只是加速访问的手段。即使链下索引服务不可用，用户依然可以直接从链上获取完整的 DID Document，数据不会丢失，身份的自主权始终掌握在用户手中。

## 注册与管理的分离

传统方案通常将注册和管理逻辑放在同一个合约中，耦合度高。

Codatta DID 将注册入口（DIDRegistrar）和核心数据管理（DIDRegistry）拆分为独立合约。DIDRegistry 只接受授权的 Registrar 进行注册操作，而 Registrar 可以有多个、各自实现不同的注册逻辑（如免费注册、付费注册、批量注册等）。这种设计使得业务模式的调整不需要触碰核心数据层。

## 跨链可移植性

传统方案的身份天然被锁定在部署链上。did:ethr 的身份只在以太坊上有效，要在其他链上使用需要重新注册一个不同的 DID。

Codatta DID 的 UUIDv4 标识符不含任何链信息，合约也可以部署到不同的 EVM 兼容链，因此在标识符设计上具备链无关性。这为未来实现跨链统一身份提供了基础。验证方法层面也已支持非 EVM 生态（如 TON），为后续扩展到更多链预留了空间。
